DNS and Connectivity Troubleshooting Lab

A structured lab note for practicing DNS lookups, basic connectivity checks, packet observation, and plain-language troubleshooting documentation.

Writeup Format

  • Objective
  • Environment

  • Findings
  • Conclusion

Objective

Practice a repeatable troubleshooting flow for a common networking problem: confirming whether a client can resolve names, reach a destination, and show useful evidence for the next step.

Environment

  • Windows client for command-line checks.
  • Wireshark or packet capture tool for observing DNS traffic.
  • Known working internet connection.
  • Public DNS name used for repeatable lookup testing.

Procedure

  • Record the client IP configuration with a command such as ipconfig.
  • Check the configured DNS server and default gateway.
  • Run a DNS lookup for a known domain and record the response.
  • Ping or otherwise test reachability to separate name resolution from connectivity.
  • Capture a DNS query and response in Wireshark and summarize what happened in plain language.

Findings

This lab focuses on collecting evidence before changing settings. The important distinction is whether the problem is DNS resolution, local network connectivity, default gateway routing, or the remote service.

Conclusion

A short, consistent process makes basic network issues easier to explain. Record the symptom, test one layer at a time, and write down the result before moving to the next hypothesis.

Lessons Learned

  • DNS failures and connectivity failures can look similar to end users.
  • Packet captures are most useful when paired with a written hypothesis.
  • Clear notes make it easier to hand off or revisit a troubleshooting case.
  • Future versions of this note should include sanitized screenshots and sample packet details.
